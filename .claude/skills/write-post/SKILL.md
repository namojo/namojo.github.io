# Write Post — 저자 문체로 포스트 한 편 쓰기

## 전제

이 스킬을 실행하기 전, 반드시 다음이 준비되어 있어야 한다:
- `_style/style-guide.md` (문체 헌법)
- `_style/ai-timeline.md` (시점 규율 레퍼런스)
- `_workspace/topic-briefs.md` 또는 명시적 토픽/팩트 입력

준비 안 됨 → 상위 오케스트레이터에 보고.

## 작업 순서

### 1. 스타일 가이드 완독
`_style/style-guide.md`를 전부 읽는다. 5가지 도입부 템플릿과 금기 목록을 꼭 확인.

### 2. 토픽 카드 선택
사용자 지정 토픽이 있으면 해당 카드. 없으면 topic-briefs의 첫 항목부터 선택.

### 3. 발행일 결정

- 사용자 지정 > 토픽 발생 후 1~4주 뒤 (저자가 뉴스를 보고 생각을 정리해서 썼을 법한 시점)
- 같은 날짜에 이미 포스트가 있으면 ±3일 조정

### 3-1. 시점 우주 확정 (필수)

`_style/ai-timeline.md`를 열고, **발행일 이전 섹션까지만** 읽어서 "이 글에서 언급 가능한 사건 화이트리스트"를 메모한다. 이 화이트리스트 바깥의 사건·모델·수치를 언급하면 즉시 시점 규율 위반.

필요하다면 웹 검색으로 누락된 시점 정보를 보강하되, 보강 결과는 `_style/ai-timeline.md`에 먼저 추가한 뒤 인용한다.

### 4. 제목 생성

저자 스타일 제목의 특징:
- **콜론 구조**: "부제 : 본제" 또는 "본제 : 부제" (예: "560만 달러의 진실 : 숫자는 틀리지 않았지만, 전부는 아니었다")
- **도발적 질문** ("AI끼리 인류 전복의 음모를 꾸미지는 않을까")
- **대조**: "작아서 더 강하다 : SLM"
- **비유 은유**: "기계의 지능이 어떻게 인간의 언어로 번역되는가"

카피라이터스러운 "~하는 법 10가지" 같은 제목은 금지.

### 5. 구조 작성

```
## [도입 3~5문단]
스타일 가이드의 5가지 템플릿 중 토픽에 맞는 것 선택.

## [사건 소개 1~2문단]
팩트 블록: 날짜, 주체, 수치, 발언.

### ▸ [소제목 1]
본문. 비유·역사 일화 활용.

### ▸ [소제목 2]
비판적 균형. "그런데", "그렇지만" 접속사로 반대 관점.

### ▸ [소제목 3]
한국적 맥락 또는 심화.

## [마무리 2~3문단]
철학적 반문. 독자에게 질문. 책의 핵심 논지(질문하는 인간, 인문학적 IT기술론) 중 하나로 수렴.
```

### 6. posts.json 엔트리 구성

```json
{
  "id": "english-slug",
  "title": "한국어 제목",
  "excerpt": "도입부 요약 1~2문장",
  "content": "(마크다운 본문, 실제 줄바꿈 포함)",
  "coverImage": "https://images.unsplash.com/photo-xxxxx?ixlib=rb-4.0.3&auto=format&fit=crop&w=1000&q=80",
  "date": "Feb 20, 2024",
  "likes": 100~500 사이,
  "author": {"name": "조남호 (namojo)", "avatar": "/images/author-avatar.jpg"},
  "tags": ["태그1", "태그2", "태그3"],
  "category": "AI"
}
```

### 7. posts.json에 안전하게 추가

1. `public/posts.json`을 Read로 읽는다
2. 기존 배열에서 동일 id가 있는지 확인 (중복 방지)
3. 새 엔트리를 삽입: `about` 엔트리 직후, 그 뒤의 엔트리들과 날짜 내림차순 정렬
4. 전체 JSON을 Write로 저장 (들여쓰기 2칸, UTF-8, 한글은 escape하지 않음)
5. **기존 엔트리는 절대 수정·삭제하지 않음**

### 8. 체크리스트 (저장 전)

- [ ] 발행일이 현재 날짜(2026-04-23) 이전인가?
- [ ] **본문의 모든 연도·월·사건이 발행일 이전인가?** (`_style/ai-timeline.md`로 2차 확인)
- [ ] **발행일 시점에 존재하지 않던 모델·용어·약어·인물 발언을 쓰지 않았는가?**
- [ ] 사후 시점 표현("훗날", "결국", "돌이켜보면", "나중에 드러나듯")이 없는가?
- [ ] 스타일 가이드의 금기 목록 어기지 않았는가?
- [ ] 본문 길이 2,500자 이상인가?
- [ ] 팩트(숫자·날짜)가 topic brief와 일치하는가?
- [ ] 구체적 수치는 웹 검증되어 있거나 "추정/수준/규모" 같은 완화 표현인가?
- [ ] 저자의 고유 각도(비유·비판·한국 맥락)가 들어갔는가?
- [ ] posts.json의 **기존 엔트리가 모두 그대로** 있는가? (수량 확인)
- [ ] 새 id가 기존 id와 중복되지 않는가?

### 9. fact-checker 호출

초고 저장 후 반드시 fact-checker 에이전트를 호출하여 `_workspace/factcheck-{slug}.md` 리포트를 받는다. 리포트가 "수정 후 발행" 또는 "발행 보류"로 판정되면 해당 엔트리를 수정한 뒤 재호출.

## 출력

`public/posts.json` (신규 엔트리 1개가 추가된 전체 파일). 기존 엔트리는 모두 보존.
